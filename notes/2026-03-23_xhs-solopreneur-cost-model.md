# 深度提取｜一人公司，所需成本

链接：http://xhslink.com/o/5IG5jdCpBaB

• 标题：一人公司，所需成本
• 关键要点（<=5）：

1. 列出单人做产品常见工具栈：Claude、Supabase、Vercel、Stripe、GitHub 等。
2. 成本叙事主轴：固定成本极低（核心支出强调 Claude 订阅 + 域名）。
3. 评论区质疑点：真实商用成本、额度限制、封号/稳定性、国内可访问性。
4. 实操风险：免费层可用于MVP，但放量后会触发迁移与付费成本。
5. 结论：更像“低成本启动清单”，不是完整经营成本表。

已补上，下面是“含图片证据块 + 图文一致性审计”的最终版：

1) 证据锚点（原文 + 图片证据）

• 标题：一人公司，所需成本
• 文本与图片共同列出工具成本清单：
Claude($20/月)、Namecheap($12/年)、Stripe(2.9%抽成) + 其余大量免费层服务（Supabase/Vercel/GitHub/Resend/Clerk/Cloudflare/PostHog/Sentry/Upstash/Pinecone）
• 互动信号高：赞 1132 / 藏 1758 / 评 207
• 评论高频反证：
• “最后收入：0”
• “云服务器还没算”
• “商用免费层不一定够用”
• “部署域名可访问性问题（区域网络）”

2) 底层机制拆解（为什么成立）

• 成立前提：MVP阶段靠免费层+AI协作可极限压低启动成本。
• 真实分歧点：从“能上线”到“可持续经营”存在隐性成本外溢：

1. 免费额度触顶后的阶梯付费
2. 可用性/网络问题带来的运维替代成本
3. AI提效依赖操作者经验，非线性复现
4. 支付抽成、获客与留存成本常被漏算

3) 结构模型 / 方法框架（可迁移）

一人公司成本四层模型

1. 启动层：域名 + AI订阅 + 免费后端/部署
2. 运行层：监控、日志、超额配额、邮件、缓存
3. 增长层：支付抽成、带宽、客服、合规
4. 商业层：获客成本、转化率、留存、复购

实操建议：预算至少按四层出月度表，不能只看“工具订阅费”。

4) 边界与反例

• 适用：技术驱动个体、验证型产品、快速试错场景。
• 不适用：高可用/强合规/跨区稳定访问要求高的业务。

5) 一句话结论

这条内容是“低门槛启动指南”，不是“完整经营成本模型”；真正难点在上线后的持续成本与商业化闭环。
