# 你不是在 Vibe Coding，你是在给未来的自己挖坑

## 内容提取（OCR整理）

这组图的核心观点：

- 纯“凭感觉先写再说”的 Vibe Coding 短期很爽，但长期容易造成上下文断裂与维护爆炸。
- 问题并非创造流动性本身，而是把“流动”误当成“无序”。
- 真正可持续的做法不是写厚重文档，而是建立一套轻量锚点系统。

### 核心心理机制

文中指出：即时反馈（界面跑起来、功能可点）会持续刺激行动；
而需求拆解、用户旅程、文档约束等前置工作缺少即时奖励，容易被大脑自动回避。

结果是：
- 先跑起来，后补文档；
- 三个月后回看代码，改一处崩多处；
- 不是在维护产品，而是在拆炸弹。

### 关键纠偏：六份文档

文中给出一套“六文件框架”用于独立项目：

1. `Master Plan.md`（宏观愿景）
- 回答：为什么做？为谁做？

2. `Implementation Plan.md`（实施计划）
- 定义构建顺序与 MVP 必须项

3. `Design Guidelines.md`（设计规范）
- 颜色、字体、组件风格、交互原则

4. `User Journey.md`（用户旅程）
- 用户从进入到完成核心动作的关键路径与流失点

5. `Tasks.md`（当前任务清单）
- 拆成可执行任务，回答“今天做什么”

6. `Rules.md` / `Agent.md`（行为准则）
- 规范 AI/开发者执行方式，减少偏航

### 文档顺序与工作模式

文中强调文档生成顺序不能乱：
`Master Plan -> User Journey -> Implementation Plan -> Design Guidelines -> Tasks -> Rules`

并强调一种关键用法：
- 不要让 AI 直接“生成文档”；
- 要让 AI 先“采访你”，通过问答把模糊直觉转成清晰判断。

### 关于 CLAUDE.md 的定位

文中给出一个实操观点：
- `CLAUDE.md` 不是普通说明文档，而是 AI 每次会话加载的“上岗训练手册”；
- 它应汇总核心文档路径、当前阶段与工作规则，提升每次任务的上下文质量。

结论：
- 文档不是限制创造力，而是为创造力降低后期成本。
- 先有清晰上下文，AI coding 才真正发挥威力。
